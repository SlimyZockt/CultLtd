- `./run.sh`

- `odin build . --debug`

- `odin build . --debug -define:PLATFORM=1`
Build optimized binary:
- `odin build . -o:speed -define:PLATFORM=1`

## Lint and Static Checks
Type check package:
- `odin check .`
Recommended style checks:
- `odin check . -vet-style -vet-tabs`
Strict checks (may fail due to existing unused declarations):
- `odin check . -vet -strict-style`
Notes:
- `-strict-style` enforces attached braces and trailing-comma style.
- Full `-vet` can fail on helper files even if game code is fine.

## Single-Test Guidance
There is no custom per-test-name runner configured.
Use one of these approaches:
1) Add/use one `@(test)` procedure in a package and run `odin test <pkg>`.
2) For file-level harness checks, run the file directly:
   - `odin run steam/test.odin -file --debug`
Repository currently relies more on (2) than formal unit tests.

## Compile-Time Config
Observed `#config` keys in main game code:
- `PLATFORM :: Platform(#config(PLATFORM, Platform.NONE))`
- `g_cult_debug := #config(CULT_DEBUG, ODIN_DEBUG)`
Platform enum values:
- `.NONE = 0`
- `.STEAM = 1`
Practical usage:
- Default local run: omit `PLATFORM` (uses `.NONE`)
- Steam networking path: `-define:PLATFORM=1`

## Code Style - Imports
Import order pattern used here:
1) base/core runtime imports
2) core modules (`fmt`, `log`, `math`, `os`, etc.)
3) vendor modules (`vendor:raylib`, stb, steamworks)
4) local modules (`./steam/`, `./aseprite`)
Example:
- `import "base:runtime"`
- `import "core:fmt"`
- `import vmem "core:mem/virtual"`
- `import rl "vendor:raylib"`
- `import steam "./steam/"`

## Code Style - Formatting
- Tabs for indentation
- 1TBS brace style
- Keep `switch` and `case` alignment clean
- Prefer trailing commas in multi-line literals
- Prefer guard clauses (`if cond do return`)
- `defer` is scope-based (not function-based); defers in loops still run correctly when `break`/`continue` exits an iteration scope
- Keep procedures focused and short where practical

## Code Style - Naming
- Types/structs/enums: PascalCase (`CultCtx`, `EntityHandle`)
- Procedures: snake_case (`update_logic`, `entity_add_sync_server`)
- Constants: SCREAMING_CASE (`MAX_PLAYER_COUNT`, `LOG_PATH`)
- Globals: `g_` prefix (`g_ctx`, `g_arena`, `g_spall_ctx`)
- Bitsets: paired `*Bits` and `*Flags`
- IDs/units: `distinct` types (`PlayerId`, `TextureId`, `Seconds`)

## Code Style - Types and Layout
- Use `distinct` for IDs and unit-like values
- Use `bit_set` for flags
- Use `#packed` for network packet structs
- Use unions where already established in module design
- Preserve binary layout when changing network data structures

## Error Handling and Logging
- Use `assert(...)` for invariants
- Use `ensure(err == nil)` for setup that must succeed
- Use `log.info/debug/warning/error` for diagnostics
- Use `log.panicf` only for unrecoverable states
- In `proc "c"` callbacks, restore `context = g_ctx` first

## Memory Management
- Arena allocator (`vmem`) is the dominant pattern
- Typical flow:
  - `vmem.arena_init_growing(...)`
  - assign allocator to `context`
  - `defer vmem.arena_destroy(...)`
- For temporary allocations:
  - `temp := vmem.arena_temp_begin(...)`
  - `defer vmem.arena_temp_end(temp)`

## Conditional Compilation and Attributes
- Steam-only code uses `when PLATFORM == .STEAM { ... }`
- Debug-only code uses `when ODIN_DEBUG { ... }`
- Common attributes: `@(thread_local)`, `@(private)`
- Do not introduce new config keys without explicit need

## Agent Workflow
Before editing:
- Read entire impacted files, not only grep snippets
- Follow existing naming, layout, and allocator patterns
- Avoid broad refactors unless requested
After editing:
- Run `odin check .`
- Run `odin check . -vet-style -vet-tabs` for style-sensitive changes
- If harness code changed, run `odin run steam/test.odin -file --debug`
Avoid:
- Inventing missing scripts (for example `build.sh`)
- Assuming a formal unit test suite exists
- Introducing style conventions that conflict with existing Odin code
